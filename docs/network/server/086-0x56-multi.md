# Multi Server (`SMulti`)

| Item | Value |
| --- | --- |
| Direction | Server to client |
| Command | `0x56` (86) |
| Transform | `static` |
| Name provenance | Microsoft C++ RTTI in the target |
| Behavioral label | Multi Server |

## Purpose

This packet replaces the alternate server list used by the title-screen server-selection flow. The list is compressed inside the otherwise decoded packet body.

The constructor calls `net_server_packet_base_ctor` with opcode `0x56` and installs the exact `SMulti` vtable. `net_server_packet_factory_ctor` registers the same opcode with this constructor.

The exact RTTI name remains `SMulti`; it should not be renamed to `SMultiServer` merely to match the friendly label.

## Exchange

[`SVersionCheck`](000-0x00-version-check.md) supplies a CRC32 for the server-list serialization. When the local list is empty or its CRC differs, `ServerSelectDialogPane` opens in request mode and sends [`CMulti`](../client/087-0x57-multi-server.md) operation `1`.

```text
server-list CRC differs
    -> CMulti operation 1 requests the list
    -> SMulti carries a compressed replacement
    -> client inflates, installs, and saves mServer.tbl
    -> dialog rows refresh
    -> one record selects automatically; otherwise the player chooses
    -> CMulti operation 0 sends the selected server ID
```

## Body

```text
packet SMulti {
    u8      opcode                    // 0x56
    u16     compressed_length
    bytes   compressed_server_list[compressed_length]
}
```

`net_deserialize_multi_server_packet` reads the big-endian length, then retains a pointer to that many bytes in the decoded packet buffer. It does not copy the compressed bytes into the packet object.

## Inflated server list

`net_apply_multi_server_list` inflates the payload with zlib into a fixed `0x800`-byte buffer. The inflated bytes use this variable-length layout:

```text
record InflatedMultiServerList {
    u8      server_count

    repeat server_count {
        u8      server_id
        bytes   ipv4_address[4]
        u16     port
        cstring server_name
    }
}
```

`cstring` here means bytes through the first `0x00`. It is local shorthand for this inflated data, not part of the ordinary packet-body notation. The name bytes are copied into a 256-byte runtime field.

The four address bytes are copied in reverse memory order so the little-endian runtime `u32` retains their network-order numeric value. `port` is read as big-endian. `server_id` becomes the value returned in `CMulti` operation `0`; it is not the row number.

The client replaces the old record array with `server_count` records of `0x281C` bytes. Each record receives the ID, IPv4 value, port, and name from this payload. Its 10,000-byte greeting field is initialized to a single space because `SMulti` carries no greeting. Greeting synchronization is handled separately by [`SStipulation`](096-0x60-stipulation.md).

## Client handling

The primary-vtable network-event handler for the exact RTTI class `ServerSelectDialogPane` accepts opcode `0x56` and calls `ui_server_select_dialog_handle_multi`. That handler:

1. inflates and parses the replacement list;
2. saves the new records to [`mServer.tbl`](../server-tables.md);
3. automatically selects row zero when the new count is exactly one;
4. refreshes the dialog and its list controls.

Automatic selection follows the same path as a click. It sends `CMulti` operation `0` with the selected record's `server_id` and updates the locally selected endpoint state.

The zlib destination is capped at `0x800` bytes. The client stores the zlib return value and produced length but does not check either before reading `server_count` and the records. The record parser also finds each name with a NUL scan rather than a separate length field. These are properties of the client parser, not additional wire fields.

The class consumes only the declared compressed slice. No trailing `SMulti` fields have been confirmed.
