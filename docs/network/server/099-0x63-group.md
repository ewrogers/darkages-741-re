# Group (`SGroup`)

| Item | Value |
| --- | --- |
| Direction | Server to client |
| Command | `0x63` (99) |
| Encoding | derived |
| Name provenance | Microsoft C++ RTTI in the target |

## Purpose

The server uses this packet to ask about an ordinary group request, return a recruiting group box, or continue a recruiting join. Each action has its own body. There is no common trailing name field.

The constructor installs the exact `SGroup` vtable. `net_deserialize_group_server_packet` reads bodies only for actions `1`, `4`, and `5`; `net_handle_group_server_packet` applies those same three actions.

## Actions

The subtype names are project-owner protocol vocabulary. The matching client independently confirms actions `1`, `4`, and `5` from their UI effects.

| Value | Name | Client behavior |
| ---: | --- | --- |
| `1` | `Ask` | Reads a character name and opens the normal group prompt, or automatically sends `CGroup` action `3`. |
| `2` | `Members` | Reads no payload and has no handler behavior in this client. The proposed name is not independently confirmed. |
| `4` | `RecruitInfo` | Reads and displays a complete recruiting group box. |
| `5` | `RecruitJoin` | Reads one name, then sends ordinary `CGroup` action `2` for that name. |

## Body

```text
packet SGroup {
    u8      opcode                    // 0x63
    u8      action

    if action == 1 {
        string8 requester_name
    }

    if action == 4 {
        string8 leader
        string8 group_name
        string8 note
        u8      minimum_level
        u8      maximum_level
        u8      maximum_warriors
        u8      current_warriors
        u8      maximum_wizards
        u8      current_wizards
        u8      maximum_rogues
        u8      current_rogues
        u8      maximum_priests
        u8      current_priests
        u8      maximum_monks
        u8      current_monks
    }

    if action == 5 {
        string8 target_name
    }
}
```

## Why the group box differs

`CGroup` action `4` publishes the form, so it sends only the five requested maximums. `SGroup` action `4` fills the viewing window, so the server adds the current membership beside each maximum. The server form therefore uses five interleaved `(maximum, current)` pairs.

The matching client does not swap Rogue and Monk. It reads five pairs in a loop, copies each pair unchanged into the corresponding UI row, and separately sums the maximum and current columns. The row order is Warrior, Wizard, Rogue, Priest, then Monk, matching the recruiting block in [`SSelfLook`](057-0x39-self-look.md).

The proposed serializer that assigns the third maximum to `MaxMonks` and the fifth maximum to `MaxRogues` has those two field names reversed. Its current-count assignments happen to remain aligned. Assigning the separately exposed `Name` property from `Leader` is only an object-model convenience; it does not describe another wire field.

## Auxiliary raw consumer

The RTTI-backed `DescPane` compares the decoded body's first byte directly. Opcode `0x63` closes the current description pane, but this route does not read `action` or any payload field. The factory-backed `SGroup` deserializer and world handler still own the body schema.

## Known limit

For action `2`, this client stops after the action byte. It neither reads a name nor updates the group UI. A server or capture is still needed to establish whether `Members` is an active subtype in this protocol version.
