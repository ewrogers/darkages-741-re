# Map (`SMap`)

`SMap` supplies a rectangular block of ground and static map cells. It does not select the map or allocate its grid. [`SMapSize`](021-0x15-map-size.md) must already have started a cache-miss transfer.

| Item | Value |
| --- | --- |
| Direction | Server to client |
| Command | `0x06` (6) |
| Transform | derived |
| Behavioral alias | Map rectangle |
| Runtime owner | `WorldPane` |
| Transfer setup | [`SMapSize`](021-0x15-map-size.md) |
| Related row format | [`SMapPart`](060-0x3c-map-part.md) |
| Name provenance | Microsoft C++ RTTI in the target |

## Plaintext body

```text
packet SMap {
    u8      opcode                    // 0x06
    u8      start_x
    u8      start_y
    u8      width
    u8      height

    repeat height {
        repeat width {
            u16 ground_tile
            u16 left_static_tile
            u16 right_static_tile
        }
    }
}
```

The complete body size is:

```text
5 + width * height * 6
```

Cells are sent in row-major order. `start_x` advances across each inner row, then `start_y` advances to the next row.

The three tile IDs match the client-confirmed six-byte [MAP cell](../../file-formats/map.md): one ground diamond followed by the two isometric static-art sides. They are big-endian on the network. `map_set_cell` converts them to the client's native in-memory words.

## Parsing and dispatch

The exact RTTI `SMap` deserializer reads the four header bytes, widens each to a 16-bit object field, and retains a pointer to the remaining cell records. It does not copy those records or store their byte count.

The live gameplay path does not use that parsed object. `net_dispatch_server_packet` checks the raw decoded body's opcode and passes the raw body to `net_handle_map_server_body`. This is deliberate and matches the raw-body handling used by `SMapPart`.

## Applying the rectangle

The handler first checks `WorldPane`'s map-transfer-active flag. [`SMapSize`](021-0x15-map-size.md) sets this flag when the local map cache is missing or its CRC does not match. If the flag is clear, `SMap` is ignored.

For an active transfer, the handler reads each cell and calls `map_set_cell` with:

```text
x = start_x + column
y = start_y + row
```

`map_set_cell` checks each coordinate against the grid allocated by `SMapSize`. An out-of-grid cell is discarded. An accepted cell replaces all three tile IDs and marks the map-cell storage dirty.

## Relationship to the other map packets

| Packet | Role |
| --- | --- |
| `SMapSize` (`0x15`) | Selects the map, dimensions, flags, weather, art mode, and cache value; sends `CMapRequest` on a cache miss |
| `SMap` (`0x06`) | Applies one arbitrary rectangle while that cache-miss transfer is active |
| `SMapPart` (`0x3C`) | Applies one complete row, updates loading progress, and completes the transfer after the final row |

`SMap` is therefore not a duplicate of `SMapSize`. One is content and the other is setup.

## Is it still used?

The client path is reachable, so `SMap` is not dead code. However, it is incomplete as a standalone map-transfer protocol. Its handler does not:

- clear the active-transfer flag;
- create or update `MapLoadingPane`;
- calculate the final CRC;
- write the local map cache; or
- call the map-transfer completion path.

`SMapPart` performs all of those completion duties when its row index reaches the last map row. No other `SMap` consumer or typed-object reader was found.

`SMap` changes can still be persisted indirectly. If a later final-row `SMapPart` completes the active transfer, its full-grid cache write includes every cell currently in memory, including surviving `SMap` rectangle changes. `SMapPart` rows received afterward may overwrite overlapping cells first. Without that later completion step, `SMap` changes remain only in memory and no cache file is written.

Given the lack of observed traffic and the missing standalone completion step, `SMap` is most likely an older compatibility path or an auxiliary rectangle update intended to run alongside another transfer message. That conclusion is an inference from the client. The exact server-side use is not confirmed.
