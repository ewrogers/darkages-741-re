# Hello (`CHello`)

The client answers the server welcome with the literal five-byte body `baram`. This is the first outgoing control message in the startup exchange, immediately before `CVersion`.

| Item | Value |
| --- | --- |
| Direction | Client to server |
| Command | `0x62` (ASCII `b`) |
| Transform | `static` |
| Name provenance | Project-owner protocol name; literal body confirmed in the target |

There is no client packet RTTI class. `CHello` is protocol vocabulary based on its role as the reply to `SHello`. The target itself supplies the older name evidence by formatting the ASCII text `baram` and submitting exactly five bytes. `baram` may be an engine codename, but the executable does not explain the word.

The ordinary outbound transform selector treats its leading `b` as opcode `0x62`, which belongs to the static-key set. The resulting wire body therefore carries the normal encrypted sequence and trailer for that transform.

## Trigger

`TerminalPane2` queues this body when either `ESC C` or `ESC S` completes the initial connection transition. It then calls `net_reset_client_packet_sequence`, builds [Version (`CVersion`)](000-0x00-version.md), and queues that raw body.

## Body

```text
control CHello {
    u8 text[5]                 // ASCII "baram"
}
```

The live decrypted capture supplied for this study ends the body with `0xEB`, but the 7.41 builder supplies `0x6D` (`m`) at that position. The capture representation therefore does not reproduce this first transformed body exactly. The client buffer and submitted length make `62 61 72 61 6D` the confirmed plaintext.

The server trigger is documented under [Hello (`SHello`)](../server/126-0x7e-hello.md).
