# Tell (`CTell`)

`CTell` sends a private message to one named character. The protocol and client call this a tell; players commonly call the same feature a whisper.

| Item | Value |
| --- | --- |
| Direction | Client to server |
| Command | `0x19` (25) |
| Transform | `derived` |
| UI owner | RTTI classes `TellReceiverInputPane` and `TellInputPane` |
| Name provenance | Project-owner protocol vocabulary, confirmed by the local builder and RTTI pane names |

## Body

```text
packet CTell {
    u8      opcode                    // 0x19
    string8 target_name
    string8 message
}
```

The connection identifies the sender, so the body needs only the recipient and message. Both strings are byte-oriented game text rather than Unicode. The common encrypted-packet sender adds its trailing zero after this meaningful body.

The server also recognizes two special recipient strings:

| `target_name` | Destination |
| --- | --- |
| `!` | Guild channel |
| `!!` | Group channel |

These are channel conventions inside the ordinary recipient field. The client does not select another opcode, add a channel byte, or otherwise change the body. It sends and remembers `!` or `!!` like an ordinary character name.

## Input flow

When no higher-priority pane consumes the key, double quote (`"`) opens `TellReceiverInputPane`. Submitting the recipient opens `TellInputPane` with this visible prefix:

```text
-> target_name: <message>
```

The prefix is local UI text and is not transmitted. `net_send_tell` reads the retained recipient and typed message, writes the two `string8` values, and submits the packet. It does not add the message to any local display or history.

The recipient pane keeps ten recently used names in 13-byte records. Each record holds up to 12 transmitted bytes plus a null terminator.

## Length limits

These limits are measured in bytes. An ASCII character uses one byte, while a Korean DBCS character normally uses two.

| Layer | Recipient | Message |
| --- | ---: | ---: |
| `string8` wire encoding | 255 bytes | 255 bytes |
| Builder copy limit | No name-specific clamp | 100 bytes |
| Normal UI submission | 12 bytes | `65 - target_name_length` bytes |

The recipient control can temporarily hold 13 bytes, but its submit path copies only 12. The message pane limits the complete visible line to 70 bytes. Its five fixed punctuation bytes are `-> ` and `: `, which gives:

```text
message_limit = 70 - length("-> ") - target_name_length - length(": ")
              = 65 - target_name_length
```

For a nonempty recipient of 1 through 12 bytes, the normal message limit is therefore 64 through 53 bytes. The builder's 100-byte read limit is not the normal player-facing limit because the pane applies the smaller prefix-dependent cap first.

The special guild target `!` permits 64 message bytes. The group target `!!` permits 63.

The builder itself does not reject zero-length strings. Higher-level server rules still decide whether the recipient is valid and whether the message may be delivered.

## Server delivery and responses

There is no local echo and no dedicated RTTI-backed `STell` class in this client. A displayed incoming whisper, sent-message confirmation, or delivery error must therefore come back through a server message path.

[`SMessage`](../server/010-0x0a-message.md) is the confirmed generic server-text mechanism and is consistent with the observed tell behavior. Its type byte decides whether text enters persistent history, the floating overlay, or another display. The exact `SMessage` type used for successful whispers and failures has not been confirmed by a paired capture, so the book does not assign a channel name to one of those values yet.

The server interprets `!` and `!!` as guild and group destinations. The server-managed Listen to Whisper setting and ignored-user list can affect direct delivery without changing the `CTell` body.
