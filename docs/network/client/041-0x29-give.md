# Give (`CGive`)

`CGive` asks the server to transfer an inventory item to another living object. The client uses this path when the player drags an item onto another player, an NPC, or a monster.

| Item | Value |
| --- | --- |
| Direction | Client to server |
| Command | `0x29` (41) |
| Transform | derived |
| Runtime owners | `InvItemPane`, `DraggedInvItemPane`, and `WorldPane` |
| Name provenance | Project-owner protocol name, confirmed against the local builder |

## Plaintext body

```text
packet CGive {
    u8      opcode                    // 0x29
    u8      slot
    u32     object_id
    u32     quantity
}
```

The body is 10 bytes including the opcode. `slot` is the one-based inventory slot retained by `InvItemPane`. `object_id` identifies the living object beneath the drag release.

`quantity` is a `u32`, not a one-byte field. `net_send_give` writes it with the same big-endian 32-bit helper used for `object_id` and submits all 10 bytes.

## Choosing the target

Dragging an inventory item creates a `DraggedInvItemPane`. Its release event is routed through the pane tree and then through `ui_world_pane_handle_living_object_message` when the selected world object is living.

The living-object handler rejects the local character's own object ID. For another living object, it sends TimerHandler event `1` and the target object ID back to the source `InvItemPane`. The living-object hierarchy covers the human and monster branches used by other players, NPCs, and monsters.

`ui_inventory_item_handle_timer` reads the source slot. A normal item reaches `ui_inventory_item_send_give` and then `net_send_give`. Slot `60` is the gold entry and takes the separate [`CGiveGold`](042-0x2a-give-gold.md) dialog path.

## Quantity behavior

The confirmed drag-to-living-object path always sends `quantity = 1`. Unlike [`CDrop`](008-0x08-drop.md), it does not open the stack-quantity prompt for ordinary items.

The packet builder still accepts and writes a full `u32` quantity. A nearby `ui_inventory_item_submit_give_quantity` callback can forward another nonzero value, but no static reference connects that callback to the live drag path. Support for giving larger item stacks therefore remains unconfirmed at the UI level.

## Server outcome

The matching success and rejection responses have not yet been traced. The body, target selection, self-target rejection, gold-slot split, and quantity-one client behavior are confirmed in the version 741 binary.
