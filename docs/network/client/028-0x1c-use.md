# Use (`CUse`)

`CUse` asks the server to activate one item in the character's inventory. The client identifies only the inventory slot. It does not say whether the item should be consumed, equipped, opened, or used in some other way.

| Item | Value |
| --- | --- |
| Direction | Client to server |
| Command | `0x1C` (28) |
| Transform | derived |
| UI owners | RTTI classes `InvItemPane`, `InventoryPane_A`, and `GUIBackPane_Interface` |
| Name provenance | Project-owner protocol name matched to the locally confirmed builder behavior |

## Body

```text
packet CUse {
    u8      opcode                    // 0x1C
    u8      slot                      // one-based inventory slot
}
```

`net_send_use_inventory_item` submits exactly these two meaningful bytes. The common client-packet submission path keeps its usual local trailing zero outside the logical body.

The slot is the same one-based value supplied by [`SAddInventory`](../server/015-0x0f-add-inventory.md). The inventory receive paths accept slots `1` through `60`, and each live `InvItemPane` retains its slot for later use. The sender reads that retained byte without adding another range check.

## Activation paths

A left-button double click inside a live `InvItemPane` activates that item.

The game interface also provides a number-key path while the item inventory mode is selected. Keys `1` through `9` activate slots `1` through `9`, and `0` activates slot `10`. The inventory helper sends nothing when the selected slot has no live item pane.

Both paths reach the same sender.

## Server-managed denial metadata

Before constructing the packet, the client copies the item's name and consults the RTTI-backed `DeniedItemList` item table. A matching entry consumes the activation inside the client, so no `CUse` packet is sent.

`DeniedItemList` is the runtime consumer, not the source of the entries. It subscribes to three hardcoded metadata names:

- `BItems`
- `BSkills`
- `BSpells`

The tables use the normal [server-managed metadata cache](../../file-formats/metadata.md). The server advertises table names and CRC values through [`SMetaData`](../server/111-0x6f-meta-data.md). If a subscribed table is missing or stale, the client requests it with [`CMetaData`](123-0x7b-meta-data.md), validates the returned compressed payload, saves it under `metafile/`, and rebuilds the in-memory denial lists.

Only the table names, parser, and lookup behavior are hardcoded. The entries can be supplied and replaced by the server. This is client-side suppression for normal play, not authoritative validation; a server still has to reject invalid use requests itself.

No other item behavior is applied locally before submission. In particular, the client does not decrement a stack, remove the inventory pane, equip the item, or update character stats in anticipation of success.

## Server-controlled result

The body has no action kind, quantity, item ID, or target. The server resolves the character's current item in `slot` and decides what using it means.

There is therefore no single required response packet. Depending on the item and result, the server can use the normal state packets, including:

- [`SRemoveInventory`](../server/016-0x10-remove-inventory.md) or another `SAddInventory` to replace the inventory slot;
- [`SAddEquip`](../server/055-0x37-add-equip.md) when the item becomes worn equipment;
- [`SStatus`](../server/008-0x08-status.md) for changed vitals, statistics, or weight;
- [`SMessage`](../server/010-0x0a-message.md) for a result or failure; and
- appearance, sound, or effect packets when the item requires them.

These are independent server packets, not a fixed acknowledgement sequence built into `CUse`. Exact response combinations remain dependent on the server and item type.

## Known limits

The current installation contains 19 cached metadata files, but none are named `BItems`, `BSkills`, or `BSpells`. No supplied capture contains those table names either. The present cache therefore provides no denial entries to enumerate, even though the download, cache, parser, and lookup paths are implemented.

No supplied capture establishes one response sequence that applies to every consumable or equippable item.
