# Say (`CSay`)

`CSay` sends text entered through the ordinary Say or Shout prompt. The body carries the public-speech mode followed by one byte-length-prefixed game string.

| Item | Value |
| --- | --- |
| Direction | Client to server |
| Command | `0x0E` (14) |
| Transform | `derived` |
| UI owner | RTTI class `ChatInputPane` |
| Name provenance | Project-owner protocol vocabulary matched to the confirmed builder |

## Body

```text
packet CSay {
    u8      opcode                    // 0x0E
    u8      speech_mode               // SpeechMode
    string8 text
}
```

[`SpeechMode`](../protocol-types.md#speechmode) defines the shared values. The confirmed interactive paths produce:

| Input | Mode | Local input prefix | Input palette |
| --- | ---: | --- | ---: |
| Enter | `0`, Say | `<character>: ` | `0xFF` |
| `!` | `1`, Shout | `<character>! ` | `0x45` |

The prefix belongs to `ChatInputPane`. Only the typed text is transmitted, so the client does not add `!` to the `CSay` string.

The input path copies at most 72 text bytes and submits only lengths greater than one byte. Text remains the client's byte-oriented game encoding rather than Unicode. The common encrypted-packet sender adds its normal trailing zero after this meaningful body.

## Spell chants

The interactive `CSay` flow does not produce Chant mode `2`. Timed spell lines use the separate [`CSpellDelaySay`](078-0x4e-spell-delay-say.md) opcode `0x4E`, which carries only `string8 text`. The server presents that spell text to nearby clients as [`SSay`](../server/013-0x0d-say.md) Chant mode.

## Server delivery

Project-owner runtime behavior establishes the server-side scope: Say is delivered within visible range and Shout is delivered map-wide. The client packet contains no range or destination field; the server chooses recipients.
