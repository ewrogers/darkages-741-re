# Drop (`CDrop`)

`CDrop` asks the server to remove a quantity from one inventory slot and place it on a chosen map tile. The client sends it when the player drags an inventory item into the world.

| Item | Value |
| --- | --- |
| Direction | Client to server |
| Command | `0x08` (8) |
| Transform | derived |
| Runtime owners | `InvItemPane`, `DraggedInvItemPane`, and `WorldPane` |
| Success | Usually `SDrawObjects` at the target tile |
| Rejection | Usually `SMessage` |
| Name provenance | Project-owner protocol name, confirmed against the local builder |

## Plaintext body

```text
packet CDrop {
    u8      opcode                    // 0x08
    u8      source_slot
    u16     x
    u16     y
    u32     quantity
}
```

The body is 10 bytes including the opcode. It identifies an inventory slot and destination tile rather than an object ID.

`source_slot` comes from the dragged `InvItemPane`. The client treats slot `60` as the gold entry and opens the separate [`CDropGold`](036-0x24-drop-gold.md) flow instead of sending `CDrop`.

## Choosing the target

Dragging an inventory item creates a `DraggedInvItemPane`. When it is released over the world, `ui_world_pane_handle_inventory_drop` uses `ui_world_pane_screen_to_tile` to convert the pointer position into map coordinates, then sends those coordinates back to the source `InvItemPane`.

The client checks that the pointer is inside the world pane and that the converted tile is inside the current map. It does not compare the target with the player's tile. A visible but distant tile can therefore reach `net_send_drop`; any shorter gameplay range is enforced by the server.

## Choosing the quantity

An item with a stored quantity greater than one opens a numeric input pane. The prompt shows a range from zero through the stored quantity. A nonzero submitted value becomes the packet's `quantity`.

For an item whose stored quantity is zero or one, the client skips the prompt and sends `quantity = 1`. Submitting zero from the quantity prompt sends nothing.

## Server outcome

On success, the server usually sends [`SDrawObjects`](../server/007-0x07-draw-objects.md) so the dropped item appears on the target tile.

When dropping is not allowed, the server usually explains the rejection with [`SMessage`](../server/010-0x0a-message.md). Project-owner behavior suggests that accepted targets are limited to roughly one or two tiles from the player. The exact cutoff, including how diagonal distance is measured, is not confirmed. The client contains no matching player-relative distance check.

The response behavior and short server range are project-owner protocol observations. The body, drag flow, quantity handling, map-bound checks, and absence of a client-side player-distance check are confirmed in the version 741 binary.
