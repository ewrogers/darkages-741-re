# Get (`CGet`)

`CGet` asks the server to move one ground item at a map coordinate into a chosen inventory slot. The client can send it after a ground-item double click or from the B-key pickup shortcut.

| Item | Value |
| --- | --- |
| Direction | Client to server |
| Command | `0x07` (7) |
| Transform | derived |
| Runtime owners | `MapItem_Pane` and `WorldPane` |
| Success | `SAddInventory` and an `SStatus` weight update |
| Rejection | Usually `SMessage`; out-of-range requests may be ignored |
| Name provenance | Project-owner protocol name, confirmed against both local builders |

## Plaintext body

```text
packet CGet {
    u8      opcode                    // 0x07
    u8      destination_slot          // normally 1..60
    u16     x
    u16     y
}
```

The packet identifies a tile, not a particular ground object. There is no object ID. If several items share the coordinate, the request itself cannot distinguish among them.

`destination_slot` is normally the first empty inventory slot. `session_find_first_empty_inventory_slot` scans slots `1` through `60` and returns `0` when the inventory is full.

## Double-click pickup

`ui_map_item_pane_handle_pointer_event` handles a left double click inside a ground item's `MapItem_Pane`. It asks for the first empty inventory slot and calls `net_send_get_from_map_item_pane` with that slot and the pane's saved world coordinates.

This path sends only when the selected slot is between `1` and `60`. A full inventory therefore prevents the double-click path from sending `CGet`.

## B-key pickup

The unmodified B key enters the pickup branch in `ui_world_pane_handle_keyboard_event`. Attempts are limited to one every 100 milliseconds.

The search order is:

1. Look for a ground item on the player's current tile.
2. If none is present, look one tile forward in the current facing direction.
3. If an item exists at either coordinate, select the first empty inventory slot and call `net_send_get`.
4. If neither tile contains a ground item, send nothing.

The object search uses broad category `8`, the category assigned to `WorldObject_Item`. The item object itself is only a local presence check. `CGet` still sends the selected coordinate rather than its ID.

Unlike double-click pickup, the B-key path does not reject slot `0` after an inventory-full result. It can submit `destination_slot = 0` and leave rejection to the server.

## Server outcome

On success, the server sends [`SAddInventory`](../server/015-0x0f-add-inventory.md) for the new inventory entry and [`SStatus`](../server/008-0x08-status.md) with the updated carried weight.

When the pickup is rejected, the server usually sends [`SMessage`](../server/010-0x0a-message.md) explaining the reason, such as the item being too heavy. A request made from outside pickup range is normally ignored without a message.

These outcome rules are project-owner protocol behavior. The request layout, input paths, search order, slot selection, and local send conditions are independently confirmed in the version 741 binary.
