# Login (`CLogin`)

`CLogin` sends the lobby account credentials together with a 16-byte installation block. The block combines persistent client values with fresh random bytes, so it changes on every login even when the account and installation stay the same.

| Item | Value |
| --- | --- |
| Direction | Client to server |
| Command | `0x03` (3) |
| Encoding | startup key |
| UI owner | RTTI class `LoginDialogPane` |
| Name provenance | Related class vocabulary matched to the locally confirmed login builder |

## Body

```c
struct CLoginBody {
    u8 opcode;                         // 0x03
    u8 name_length;
    u8 name[name_length];
    u8 password_length;
    u8 password[password_length];
    CLoginInstallationBlock client;
};

struct CLoginInstallationBlock {
    u8 random_a;
    u8 encoded_random_b;
    u32be encoded_installation_id;
    u16be encoded_derived_id;
    u32be encoded_nonce;
    u16be encoded_integrity;
    u8 constant_one;                   // 1
    u8 trailing_zero;                  // supplied by net_submit_client_packet
};
```

The two strings are byte strings with one-byte lengths. The password control accepts at most eight bytes.

The builder supplies every field through `constant_one`. The common submission helper appends the final zero and includes it in the encrypted payload. A decoded trace may omit that helper-added byte when it reports only meaningful packet data.

## Installation block

The client seeds its C runtime random generator from the current time for each login. It takes two random bytes, `random_a` and `random_b`, then writes:

```c
encoded_random_b = (random_a + 0x3B) ^ random_b;
```

The server can therefore recover `random_b` and use it to reverse the remaining masks. Each integer is masked byte by byte in the client's little-endian memory and then written to the packet in big-endian order.

| Wire field | Source | Mask for local byte `i` |
| --- | --- | --- |
| `encoded_installation_id` | Persistent 32-bit installation ID | `random_b - 0x76 + i` |
| `encoded_derived_id` | Persistent 16-bit derived ID | `random_b + 0x5E + i` |
| `encoded_nonce` | Fresh 32-bit random value | `random_b + 0x73 + i` |
| `encoded_integrity` | Integrity value described below | `random_b - 0x5B + i` |

Implementations should not apply these masks directly from left to right over the big-endian wire bytes. The client masks the x86 memory representation first, so the byte order is reversed when the integer is written.

## Integrity value

`net_calculate_login_block_integrity` covers the first 12 installation-block bytes, ending with `encoded_nonce`. It uses a 256-entry table generated from polynomial `0x1021`, but its update differs from the usual CRC-CCITT formula:

```c
u16 login_integrity(const u8 *data, int length)
{
    u16 value = 0;

    for (int i = 0; i < length; i++) {
        value = table[value >> 8] ^ (value << 8) ^ data[i];
    }

    return value;
}
```

The result is masked separately and written as `encoded_integrity`.

## Persistent client values

`net_configure_default_endpoint` also prepares the two installation values used by this packet:

- The 32-bit installation ID is loaded from `HKCR\NXKRI.Ctrl.1`, value `CLSID`, or randomly created when the value is missing or invalid.
- `app_derive_installation_id16` runs the same custom recurrence over the installation ID's four little-endian bytes.
- A disguised form of that derived value is stored under `HKCR\KRIHC.Ctrl.1`, also as `CLSID`.

The relevant configuration state is:

```c
struct LoginInstallationState {
    u32 installation_id;               // app_config + 0x424
    u32 derived_id;                    // app_config + 0x428, low 16 bits used
    u32 last_encoded_nonce;            // app_config + 0x430
    u8 mscfg_marker_present;           // app_config + 0x668
};
```

The login builder stores the encoded nonce back at `+0x430`. Its later purpose remains unresolved.

If the `Mscfg.dll` marker flag is set, the builder calls the invalid-memory clearing path before constructing the packet. Normal `CLogin` submission cannot continue from that state. See [`SBadGuy`](../server/074-0x4a-bad-guy.md) for how the marker is created and [Configuration](../../application/configuration.md) for how it also changes the endpoint.

## Sent by the login dialog

`LoginDialogPane` loads `_nlogin.txt` and attaches controls in this order:

| Action ID | Control |
| --- | --- |
| `0` | `OK` |
| `1` | `Cancel` |
| `2` | `Name` |
| `3` | `Password` |

`ui_login_dialog_handle_action` reads controls 2 and 3 when action 0 is selected and calls `net_send_login_request`. After submission, the sender copies the name into `session_character_name`. Session setup later uses that name to build the MD5 salt source for derived-key packets.

## Supplied successful-login trace

The first client record in the supplied trace has two valid length-prefixed credential fields followed by exactly 15 meaningful installation bytes. Without retaining or displaying the credentials or recovered installation values, the remaining bytes validate all of these relationships:

- `encoded_random_b` recovers the mask byte used by the other fields.
- The recovered 16-bit ID matches the client's derivation from the recovered 32-bit installation ID.
- The integrity value matches the first 12 installation bytes.
- The final meaningful byte is `1`.

The trace presentation omits the common helper-added zero. The client still includes that byte before applying the startup-key transform.

## Result

The paired response is [Lobby Account Result (`SLoginCheck`)](../server/002-0x02-login-check.md). Status `0` begins game-session setup. A nonzero result displays the server's declared message.

The server-side policy for the persistent identifiers is not visible in this executable.
