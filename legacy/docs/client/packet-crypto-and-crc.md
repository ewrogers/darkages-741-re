# Packet encryption, decryption, and CRC reconstruction

This document describes algorithms recovered solely from the version-741 client and its IDA database. No online or third-party protocol implementation was used as evidence. The accompanying [minimal C99 reference](./packet-crypto-reference.c) keeps the client-specific logic easy to read and translate.

## Layer order

Outgoing binary packets pass through:

```text
builder body: [opcode, payload...]
    | net_submit_client_packet
optional dialog wrapper for 0x39 / 0x3A
    |
append/include queued NUL sentinel
    | net_encrypt_client_packet_body unless raw
common XOR transform + ordinal + integrity/seed trailer
    | net_send_client_packet
[0xAA, u16be transformed_body_size, transformed_body]
```

The receive path removes the `0xAA/u16be` frame first, then selects raw, primary-key, or derived-key handling from the server opcode.

## Mode selection

### Client to server

| Mode | Opcodes |
|---|---|
| raw | `0x00`, `0x10`, `0x48` |
| primary-key mode | `0x02`, `0x03`, `0x04`, `0x0B`, `0x26`, `0x2D`, `0x3A`, `0x42`, `0x43`, `0x4B`, `0x57`, `0x62`, `0x68`, `0x71`, `0x73`, `0x7B` |
| derived-key mode | every other opcode reaching the common sender |

`0x39` and `0x3A` both receive the dialog wrapper first. After wrapping, `0x39` uses derived-key mode and `0x3A` uses primary-key mode.

### Server to client

| Mode | Opcodes |
|---|---|
| raw | `0x00`, `0x03`, `0x40` |
| primary-key mode | `0x01`, `0x02`, `0x0A`, `0x56`, `0x60`, `0x62`, `0x66`, `0x6F` |
| derived-key mode | every other server opcode |

This switch is local evidence that the transport/decrypt layer recognizes lobby opcodes `0x01` and `0x02`, even though no RTTI packet factories are registered for them.

## Primary and derived keys

The default primary key is the nine-byte ASCII string:

```text
UrkcnItnI
```

The client can replace this primary key at runtime. The default block size and key period are nine bytes.

Derived-key mode uses a 1024-byte table constructed from `player_character_name` at `0x73D910`. The client builds it after the character name has been established:

```text
table = md5_hex(md5_hex(character_name))
repeat 31 times:
    table += md5_hex(table)
```

`md5_hex` means standard MD5 formatted as 32 lowercase ASCII hexadecimal characters. The final table is therefore 1024 **ASCII bytes**, not 512 decoded hexadecimal bytes.

This character-name MD5 expansion is distinct from the outgoing per-packet MD5 integrity tag. The former establishes the derived encryption table; the latter hashes each already-encrypted packet body.

For each encrypted packet, a nine-byte key is selected:

```text
for i = 0..8:
    index = (seed16 + (seed8*seed8 + 9*i)*i) & 0x3FF
    packet_key[i] = table[index]
```

## Server-selected salt/mask table

The raw `SVersionCheck 0x00`, subtype `0`, supplies:

```text
u8  opcode = 0x00
u8  subtype = 0x00
u32be unknown/check value
u8  salt_table_seed       // valid values 0..9
u8  primary_key_length
u8  primary_key[primary_key_length]
```

The client queues the salt selector and key update to the communications worker. The key replaces the default primary key. The observed protocol normally uses nine bytes, but the native setter accepts the transmitted length.

`net_build_packet_salt_table` at `0x568650` generates **256 logical byte values** at runtime from one of ten hardcoded formulas. It does not load ten literal 255-byte arrays. For optimized XOR operations, each logical byte is stored as a dword containing that byte four times.

Let `i` range from `0` through `255`, and let `S[i]` be the generated byte:

| Seed | Generated `S[i]` |
|---:|---|
| `0` | `i` |
| `1` | `0x80 + ceil(i/2) * (+1 when i is even, -1 when i is odd)` |
| `2` | `0xFF - i` |
| `3` | even: `0xFF - i/2`; odd: `(i+1)/2` |
| `4` | `(i / 16)^2` |
| `5` | `(2*i) mod 256` |
| `6` | `0xFF - ((2*i) mod 256)` |
| `7` | `i <= 127 ? 0xFF - 2*i : 2*i - 0x100` |
| `8` | `i <= 127 ? 2*i : 0x1FF - 2*i` |
| `9` | `0xFF - (trunc((i-0x80)/8)^2 mod 256)` |

This selector is separate from the per-packet random `seed16` and `seed8` values used to select nine characters from the character-name-derived 1024-byte table.

## Common XOR transform

Let `ordinal` be the rolling sequence byte and `block_size` normally be 9.

For every payload byte:

1. XOR with the repeating selected key.
2. Divide the payload into nine-byte blocks. For block index byte `b`, XOR the block with `S[b]`, except when `b == ordinal`.
3. XOR the entire payload with `S[ordinal]`.

Because all three stages are XOR, decryption applies the same operations. The native decoder performs them in reverse order. Selector `0` produces the identity table `S[i] = i`, which is why directly XORing the raw block index and ordinal is correct only for seed `0`.

The outgoing ordinal starts/reset at zero and increments modulo 256 for every commonly encrypted CPacket.

## Client to server encrypted body

The client generates both seeds from one `rand_s` result:

```text
seed16 = ((random & 0xFFFF) % 0xFEFD) + 0x0100
seed8  = (((random & 0x00FF0000) >> 16) % 0x9B) + 0x64
```

The transformed body is:

```text
u8 opcode
u8 ordinal
bytes encrypted_payload
u8 md5[13]
u8 md5[3]
u8 md5[11]
u8 md5[7]
u8 (seed16.low  XOR 0x70)
u8 (seed8        XOR 0x23)
u8 (seed16.high XOR 0x74)
```

The MD5 input is exactly:

```text
[opcode, ordinal, encrypted_payload]
```

The four selected digest bytes are an integrity tag, not a CRC.

## Server to client encrypted body

The local decoder expects:

```text
u8 opcode
u8 ordinal
bytes encrypted_payload
u8 (seed16.low  XOR 0x74)
u8 (seed8        XOR 0x24)
u8 (seed16.high XOR 0x64)
```

The direction-specific mask constants differ from the outgoing constants. The client decoder treats every byte between the ordinal and final three seed bytes as ciphertext and does not extract or verify a four-byte MD5 tag.

## Dialog wrapper for 0x39 and 0x3A

Before common encryption, the builder body `[opcode, payload...]` becomes:

```text
u8 opcode
u8 random1
u8 ((random1 + 0xD3) XOR random2)
u8 encoded_length_high
u8 encoded_length_low
u8 encrypted_crc_high
u8 encrypted_crc_low
bytes encrypted_original_payload
u8 0
```

Where:

```text
protected_length = original_body_length + 1

encoded_length_high =
    high(protected_length) XOR (random2 + 0x72)

encoded_length_low =
    low(protected_length) XOR (random2 + 0x73)

crc = packet_crc16(original_payload_without_opcode)

for i = 0 .. protected_length-1:
    protected_region[i] XOR= random2 + 0x28 + i
```

The protected region begins at the CRC high byte and contains the two CRC bytes followed by the original payload.

## Custom CRC-16 recurrence

The two 512-byte tables at `0x6D15E8` and `0x6D2D98` are byte-for-byte identical. They are generated with polynomial `0x1021`, but the update rule is:

```text
crc = table[crc >> 8] XOR (crc << 8) XOR value
crc &= 0xFFFF
```

| Property | Value |
|---|---|
| table polynomial | `0x1021` |
| initial value | `0x0000` |
| final XOR | none |
| table/reflection style | non-reflected, high-byte table |
| byte injection | XORed after lookup, not into the lookup index |
| check value for ASCII `123456789` | `0xBEEF` |

Because standard CRC-16/XMODEM indexes with `(crc >> 8) XOR value`, this implementation should not be labeled XMODEM even though it uses the same `0x1021` table.

## CRC uses

### Dialog packets

`net_calculate_dialog_crc16` starts at zero and feeds every original payload byte after the opcode. The result is stored big-endian in the dialog wrapper before that region is XORed.

### SRequestCRC / CReplyCRC

The client reads the server challenge as `u16be`, then deliberately feeds:

```text
low byte of challenge
high byte of challenge
```

starting from zero. It returns the result as `u16be` in `CReplyCRC 0x45`.

### Map checksum

`net_calculate_map_crc16` walks map records in row-major order. Each record contains three 16-bit values, and each word is fed low byte then high byte. The resulting 16-bit value is cached locally and transmitted in the three-byte checksum field of `CMapCRC 0x05`.
